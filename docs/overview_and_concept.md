# Обзор и концепция

## 1. Что это

`blocksnet-mcp` — **локальный MCP-сервер**, который делает агента городской аналитики `BlocksNetAgent`
доступным как MCP-инструмент. Сервер выставляет по протоколу **MCP** инструмент
`analyze_urban_question(...)`: на вход — задача на естественном языке, на выход — **строгий
машиночитаемый JSON** с гипотезами, измеренными эффектами `before→after`, кандидатными кварталами,
уверенностью и ограничениями.

> **Текущий объём.** Сервер работает **локально, на ваших данных** в `data/`
> (`blocks_with_services.gpkg`, `acc_mx.pickle`, нормативы). Транспорт — **stdio** (запуск как
> подпроцесс локальным MCP-клиентом; без сетевых портов и авторизации). Интеграция с UrbanDB и
> сетевой деплой — **вне объёма** (см. §5, помечено «Future»).

## 2. Зачем именно MCP-обёртка целиком (Вариант 2)

Рассматривались два пути:

| Вариант | Суть | Решение |
|---|---|---|
| **1. Разобрать на инструменты** | выставить 33 внутренних инструмента `blocksnet` как отдельные MCP-tools, рассуждение собирает потребитель | **отклонён** — теряется рассуждающий слой (PTR-цикл, инварианты, измеренные сценарии), который и есть ценность агента |
| **2. Обернуть агент целиком** | один MCP-инструмент `analyze_urban_question`, внутри — полный `BlocksNetAgent.run()` | **выбран** — интеллект агента сохраняется; клиент получает готовый структурированный анализ |

## 3. Что наследуется от `BlocksNetAgent` (без изменений)

Ядро переносится из репозитория `blocksnet-agent` **как есть** (пакет `blocksnet_agent/` целиком):

- **PTR-цикл гипотез** (`predict → test → revise`): 2–5 фальсифицируемых гипотез до расчётов, затем
  классификация исхода (`supported` / `refuted` / `inconclusive`).
- **RAG по инструментам** (`find_tools` / `get_tool_help`): короткие описания + полные карточки по запросу.
- **Слой инвариантов** (M1–M3, C1/C2/C3, C-Hyp): домен-нейтральные проверки свойств ответа с реентри.
- **Измеренные предложения развития**: TPE-оптимизатор зон + сценарная проверка `before→after`.
- **Формат вывода**: секции `ANALYSIS PLAN / RESULT / REFLECTION / HYPOTHESES / …`, авто-скоринг
  `confidence`, `limitations`.

Агент уже умеет читать локальную модель из `data/` (через `ensure_blocks` / `ensure_acc_mx`) — поэтому
для локального режима никакого адаптера данных не требуется.

## 4. Что добавляется в этом репозитории

- **MCP-обёртка** `blocksnet_mcp/` (`server.py`, `tools_mcp.py`, `serialize.py`, `settings.py`) —
  см. [architecture.md](architecture.md).
- **Сериализация** `AgentResult → строгий JSON` вместо длинного текста.
- Тонкий сервисный слой: запуск по stdio, чтение конфигурации, вызов `run()`.

## 5. Вне текущего объёма (🕘 Future)

Эти пункты осознанно отложены — нужны только при выходе на MAS-платформу:

- **Контекст UrbanDB** (`scenario_id`/`project_id` → данные из Urban API вместо локальных файлов) —
  адаптер `context.py`. Сейчас не делаем; справка сохранена в локальном gitignored-документе
  `docs/mas_integration_reference.md`.
- **Сетевой деплой** в ИТМО (HTTP-транспорт, Bearer-авторизация, Docker, регистрация в реестре
  «Urban services») — локальный gitignored-документ `docs/mas_registration.md`.

## 6. Границы применимости (локально)

- **LLM — внешний** (OpenAI-совместимый API: OpenRouter / lite-llm / локальный). GPU не нужен.
- **Данные — ваши локальные** из `data/`; вопросы за пределами модели агент честно помечает в `limitations`.
- **Долгий вызов:** один прогон 60–120 с (с TPE дольше) — учесть таймаут MCP-клиента и `max_iterations`.
